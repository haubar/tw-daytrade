## 測試方法

### 1. 回歸測試
重跑 Phase 1、Phase 2 開發過程中累積的既有單元測試，確認這次 QA 過程中的任何修改都沒有破壞既有功能。

### 2. 靜態程式碼檢查
- 全部 `.mjs` 檔案語法檢查（`node -c`）
- 搜尋 TODO/FIXME/殘留的 `console.log`（結果：無殘留）
- 逐一確認每個檔案的 import 是否有實際被使用（結果：無未使用的 import）

### 3. 端對端整合測試（`_test-integration-scan.mjs`）
**這是這次 QA 新增、之前完全沒做過的測試層級。** 之前每個模組（`normalize`／`history`／`factors`／`screen`／`institutional`／`storage`）都只被個別測試過，從沒有人真正完整執行過一次 `scan.mjs` 的實際 handler，驗證這些模組接在一起是否真的能動。

做法：用假的 `global.fetch` 攔截所有對外請求（TWSE、TPEx、歷史資料、三大法人），提供符合真實格式的假資料，直接呼叫 `scan.mjs` 的 `export default` handler（跟 Netlify 實際呼叫的方式完全一樣），驗證完整流程從頭到尾的正確性，包括：
- HTTP 狀態碼、回應結構完整性
- 多方/空方觀察榜排序是否符合預期（用刻意設計成「明顯強勢」跟「明顯弱勢」的假股票驗證）
- 歷史資料去重機制是否正常運作
- 各資料來源的狀態回報是否正確
- **Netlify Blobs 在沒有真實環境時的降級行為**（這個測試環境本來就沒有真實 Blobs 可用，驗證的重點是「這個必然會發生的失敗有沒有被優雅捕捉」，而不是讓整個請求跟著掛掉）

結果：14 個案例全過，包括驗證 Blobs 寫入失敗時 `storageWarning` 有正確出現、且整體請求仍回傳 200。

### 4. 前後端資料結構一致性測試（`_test-schema-consistency.mjs`）
**這是這次 QA 發現真正問題的測試。**

`src/sampleData.js`（前端本機開發用的範例資料）是手動編造的，跟 `scan.mjs` 的真實輸出是兩份各自維護的東西。用跟整合測試相同的假資料跑一次真實的 `scan.mjs`，把輸出的欄位結構跟 `sampleData.js` 逐層比對（頂層欄位、`dataSourceStatus` 子欄位、`longWatchlist`/`shortWatchlist` 裡每檔股票物件的欄位）。

**發現的問題**：`screen.mjs` 的真實輸出裡，每檔股票物件多了一個 `hasHistory` 欄位，這是內部用來過濾「沒有歷史成交量資料的新股」的暫時標記，但因為程式碳用 `{...c}` 展開整個候選股物件時沒有把它拿掉，不小心外洩到最終輸出，而 `sampleData.js` 沒有這個欄位（因為是手動編的，不知道有這個內部細節外洩）。

**修正**：`screen.mjs` 篩選完候選股之後，用解構賦值把 `hasHistory` 從物件裡拿掉，才繼續往下算分數，確保這個純內部實作細節不會出現在對外的資料結構裡。

**這類問題的風險**：如果沒有這個交叉驗證，前端在本機開發時用範例資料測試一切正常，但部署後接上真實 `scan.mjs` 輸出，雖然不會直接壞掉（多一個沒用到的欄位不影響現有元件渲染），但長期而言容易讓「範例資料」跟「真實資料」的結構持續飄移，之後改欄位時更容易出錯而不自知。

### 5. 邊界案例測試
- **全部資料源都失敗**（TWSE、TPEx、三大法人都連不上）：驗證 `scan.mjs` 回傳明確的 HTTP 500 跟錯誤訊息，而不是回傳一個「看起來正常但其實是空的」`longWatchlist`，避免使用者誤以為「今天真的沒有任何強勢股」
- **TPEx 欄位對不上**（目前最大的實際風險，`TPEX_FIELD_CANDIDATES` 還沒被真實資料驗證過）：模擬 TPEx 回傳完全預期外的欄位名稱，驗證整個掃描仍能只用 TWSE 資料正常產出結果，且 `dataSourceStatus.tpex` 會清楚標示失敗原因（包含實際欄位名稱，方便之後對照修正）

### 6. 前端建置驗證
- `npm run build` 確認 Vue + Tailwind 專案正確編譯
- 檢查編譯後的 CSS 確實包含 `@theme` 產生的工具類別
- Dev server 啟動後逐一確認所有元件模組（包含這次 QA 前所有階段新增的檔案）都能被正確載入，無匯入錯誤

以下項目在這次 QA 過程中**沒有**驗證，因為需要連到真實的 TWSE/TPEx/Netlify 環境，這個開發環境的網路權限與沙盒限制做不到，維持在 [DEPLOY_CHECKLIST.md](./DEPLOY_CHECKLIST.md) 規劃的「部署後由使用者驗證」範疇：

- T86（三大法人）端點的 `date` 參數在真實環境下是否可靠

- Netlify Scheduled Functions 的排程是否被正確註冊與觸發
- 前端實際視覺呈現（配色、排版、響應式效果）——這個環境沒有瀏覽器可以截圖確認
